## Step 3: Root Cause Analysis

### The Causal Chain

```
pubspec.yaml: drift_dev: ^2.18.0
                    ↓ pub resolves to latest matching
pubspec.lock: drift_dev: 2.31.0
                    ↓ depends on
              build: ^4.0.0  (resolved: 4.0.6)
                    ↓ build 4.x changed compilation strategy
              build_runner compiles entrypoint with 'dart compile'
                    ↓ dart compile scans all transitive dependencies
              finds sqlite3_flutter_libs: 0.6.0+eol
                    ↓ which depends on
              jni: 1.0.0
                    ↓ which declares a native build hook
              hook/build.dart  (Dart native asset build hook)
                    ↓
              'dart compile' REFUSES: "does not support build hooks"
                    ↓
              BUILD FAILS ❌
```

### Why This Happened

**Dart/Flutter native assets** is a new feature (Dart 3.3+) allowing packages to compile native code. Packages that use native code (like `jni`) declare a `hook/build.dart` file called a "build hook."

**`build` package v4.0.x** changed how it compiles the build_runner entrypoint script: it switched from `dart run` (which tolerates build hooks) to `dart compile exe` (which rejects build hooks because they require the Dart build system, not the Dart compiler directly).

**The conflict**: `drift_dev 2.31.0` needs `build 4.x`, which uses `dart compile`. But `sqlite3_flutter_libs 0.6.x` brings in `jni 1.0.0`, which has a build hook. `dart compile` cannot handle that hook → crash.

## Step 6: Why `flutter pub run` vs `dart run` vs `dart build`

| Command | Use Case | This Project |
|---|---|---|
| `dart run build_runner build` | Pure Dart projects, no Flutter SDK | ❌ Wrong — Flutter packages need Flutter context |
| `flutter pub run build_runner build` | Flutter projects with code generation | ✅ **Correct** |
| `dart build` | Compiling executables / native assets | ❌ Wrong — does not run source generators |

**Use `flutter pub run build_runner build`** because:
- The project uses Flutter packages (`path_provider`, `drift/native.dart`) that require Flutter SDK resolution
- `flutter pub run` ensures the Flutter SDK is on `$PATH` and package overrides are applied
- `dart run` may not resolve Flutter SDK packages correctly in some setups

---

## Step 7: Dart 3.8+ Build Hooks Explanation

Dart 3.3 introduced **native assets** (build hooks). Packages with `hook/build.dart` compile native code at build time.

The `build` package v4.x changed the build script entrypoint compilation:
- `build` 2.x / 3.x: uses `dart run` → tolerates packages with build hooks
- `build` 4.x: uses `dart compile exe` → rejects packages with build hooks

The **only safe path** until `build_runner` supports build hooks natively is to keep `drift_dev` on a version that requires `build <4.0.0`, and use a version of `sqlite3_flutter_libs` that does not pull in `jni` (i.e., `0.5.x`).
